# Пайплайн создания клиентских списков

Идеальная последовательность от источников данных до работающего лида в CRM и AI Sales.

---

## Общая схема

```
[1. Определение целевого сегмента]
         │
[2. Сбор сырых данных об объектах]
         │
[3. Идентификация ЛПР]
         │
[4. Обогащение контактных данных]
         │
[5. Нормализация и дедупликация]
         │
[6. Квалификация и скоринг]
         │
[7. Экспорт]
    ├─→ [7a. Bitrix24 CRM]  — контроль качества, ручная работа менеджера
    └─→ [7b. AI Sales Factory] — автоматический CHAMP-скоринг и nurturing
         │
[8. Обратная связь и итерация]
```

---

## Этап 1. Определение целевого сегмента

**Цель:** Сузить область парсинга до конкретного ICP (Ideal Customer Profile).

**Для КП (приоритет 1):**
- Регион: начать с Москвы и МО, затем расширять
- Размер: >30 домов (оптимальный объём для окупаемости PASS24)
- Статус: построен и заселён (не стройка)
- Управление: есть ТСН/СНТ или УК (не самоуправление жителей)
- Охрана: есть КПП и/или охрана (признак бюджета на безопасность)

**Нецелевые объекты (фильтровать сразу):**
- Строящиеся объекты
- Объекты без управляющей организации
- Дачные кооперативы без постоянного проживания
- Объекты, уже использующие PASS24 (проверка по существующей базе Bitrix24)

---

## Этап 2. Сбор сырых данных об объектах

**Цель:** Собрать базовую информацию об объектах целевого сегмента.

| Источник | Что собираем | Приоритет |
|----------|-------------|-----------|
| 2GIS | Название КП, адрес, телефон УК/ТСН, категория, рейтинг | Высокий |
| Яндекс Карты | Геоданные, отзывы, контакты организаций на территории | Высокий |
| Сайты-каталоги КП | Описание, количество домов, инфраструктура, контакты | Средний |
| Форумы и группы жителей | Упоминания УК, председателей, проблем с доступом | Низкий |

**Результат этапа:** Список объектов с базовой информацией (название, адрес, тип, размер).

---

## Этап 3. Идентификация ЛПР

**Цель:** Для каждого объекта найти конкретного человека с правом подписи договора.

| ЛПР | Как найти | Источники |
|-----|-----------|-----------|
| Председатель ТСН/СНТ | По ИНН/ОГРН организации | ЕГРЮЛ (nalog.ru), выписки |
| Директор УК | По ИНН/ОГРН УК | ЕГРЮЛ, сайт УК |
| Управляющий КП | Контакты на сайте КП | Сайт объекта, 2GIS |

**Алгоритм:**
1. По названию объекта → найти ТСН/СНТ/УК в ЕГРЮЛ
2. По ИНН/ОГРН → получить ФИО руководителя
3. По сайту объекта → найти контактные данные (email, телефон)
4. Перекрёстная проверка: данные ЕГРЮЛ + данные с сайта совпадают?

**Важно:** Председатель ТСН/СНТ — это ЛПР, решение которого НЕ требует согласования с собранием собственников. Это ключевое преимущество КП перед ЖК (где решение может принимать общее собрание).

---

## Этап 4. Обогащение контактных данных

**Цель:** Максимально заполнить контактную карточку ЛПР.

**Данные для сбора:**
- ФИО (из ЕГРЮЛ)
- Должность (Председатель / Директор / Управляющий)
- Телефон (с сайта, из 2GIS, из карт)
- Email (с сайта, email-домен организации, через Hunter.io / аналоги)
- Сайт организации
- ИНН и ОГРН (для верификации)

**Дополнительные данные для CHAMP-скоринга (в metadata):**
- `object_size` — количество домов/квартир (влияет на Money)
- `has_security` — наличие охраны/КПП (влияет на Challenges)
- `has_skud` — наличие существующей СКУД (влияет на Technical Readiness)
- `management_type` — ТСН/СНТ/УК (влияет на Authority)
- `region` — регион (влияет на Priority для менеджеров)

---

## Этап 5. Нормализация и дедупликация

**Цель:** Получить чистый список без дублей с единым форматом.

**Нормализация:**
- Телефоны → `+7XXXXXXXXXX`
- Email → lowercase, trim пробелов
- ФИО → "Фамилия Имя Отчество" (или "Фамилия И.О." если полные данные недоступны)
- Адреса → единый формат: "Регион, Город/Район, Название объекта"

**Дедупликация (приоритет ключей):**
1. По ИНН организации (самый надёжный)
2. По email ЛПР
3. По телефону ЛПР
4. По названию объекта + адресу (fuzzy matching)

**При слиянии:** данные из более надёжного источника (ЕГРЮЛ > сайт > 2GIS > карты) имеют приоритет.

---

## Этап 6. Квалификация и скоринг

**Цель:** Отсеять нецелевые записи и приоритизировать список.

### Quality Score (полнота данных): 0–1

| Поле | Вес |
|------|-----|
| Email ЛПР | 0.25 |
| Телефон ЛПР | 0.20 |
| ФИО ЛПР | 0.15 |
| Размер объекта | 0.10 |
| ИНН/ОГРН | 0.10 |
| Наличие СКУД (известно) | 0.10 |
| Наличие охраны (известно) | 0.10 |

**Минимальный порог для экспорта:** quality_score ≥ 0.4 (минимум email или телефон + ФИО).

### Pre-CHAMP (предварительная оценка перспективности)

| Признак | Баллы | Логика |
|---------|-------|--------|
| КП с охраной и >50 домов | +30 | Идеальный профиль |
| Есть существующая СКУД | +15 | Знают проблематику, могут быть недовольны |
| Нет СКУД, но есть охрана | +20 | Бюджет на безопасность есть, потребность явная |
| ТСН/СНТ (не УК) | +10 | Быстрое решение, один ЛПР |
| МО и крупные города | +10 | Выше бюджеты и осведомлённость |
| <30 домов | -10 | Низкая экономика |
| Нет контактных данных | -20 | Невозможно начать коммуникацию |

## Этап 7a. Экспорт в Bitrix24 CRM

**Когда использовать:** Первые партии лидов (для ручной проверки качества), крупные bulk-импорты, работа менеджеров вручную.

### Формат CSV для импорта сделок

```csv
Название сделки,Контакт,Телефон,Email,Компания,Источник,Комментарий
"КП Сосновый бор — подключение PASS24","Иванов Иван Иванович","+79001234567","ivanov@sosnovybor.ru","ТСН Сосновый бор","parser","КП 120 домов, МО, есть охрана, нет СКУД"
```

### Маппинг полей на Bitrix24

| ParsedContact | Поле Bitrix24 | Примечание |
|---------------|---------------|------------|
| `object_name` | Название сделки | Формат: "{object_name} — подключение PASS24" |
| `contact_name` | Контакт → ФИО | Создаётся связанный контакт |
| `contact_phone` | Контакт → Телефон | |
| `contact_email` | Контакт → Email | |
| `org_name` | Компания → Название | Создаётся связанная компания |
| `org_inn` | Компания → ИНН | |
| `contact_role` | Контакт → Должность | |
| `source` | Источник | "parser" |
| Всё остальное | Комментарий к сделке | Размер, наличие СКУД, охрана |

### Привязка к воронке

- Воронка: **Облако (Cloud SaaS)**
- Стадия: **Новая** (первая стадия воронки)
- Ответственный: назначается по региону или round-robin

### Способы импорта

1. **Ручной CSV-импорт** — через интерфейс Bitrix24 (Сделки → Импорт). Для начального этапа.
2. **REST API** — через `BITRIX24_WEBHOOK_URL` (тот же endpoint, что использует AI Sales). Для автоматизации.

## Этап 7b. Экспорт в AI Sales Factory

**Когда использовать:** После валидации качества данных через Bitrix24. Для автоматической обработки AI-агентами.

### Формат webhook

```json
POST /webhooks/lead-form
Content-Type: application/json

{
  "source": "parser",
  "form_id": "pass24-parser-import",
  "contact_name": "Иванов Иван Иванович",
  "phone": "+79001234567",
  "email": "ivanov@sosnovybor.ru",
  "company_name": "ТСН Сосновый бор",
  "metadata": {
    "object_name": "КП Сосновый бор",
    "object_type": "kp",
    "object_address": "МО, Одинцовский район",
    "object_size": 120,
    "has_security": true,
    "has_skud": false,
    "management_type": "tsn",
    "org_inn": "5032123456",
    "org_ogrn": "1125032001234",
    "contact_role": "Председатель ТСН",
    "quality_score": 0.85,
    "pre_champ_score": 65
  }
}
```

### Что происходит после отправки

1. **Lead Generator** (агент 5) создаёт лид в PostgreSQL, дедуплицирует по email
2. **Lead Scorer** (агент 6) запускает CHAMP-скоринг через LLM + RAG:
   - Данные из `metadata` обогащают скоринг (размер объекта → Money, наличие СКУД → Technical Readiness)
   - Результат: score 0-100
3. **Маршрутизация:**
   - score ≥ 70 (hot) → **Outreach Agent** генерирует персональное КП за минуты
   - score 40-69 (warm) → **Nurturing Agent** запускает email-последовательность (3 письма: день 1/4/10)
   - score < 40 (cold) → **Nurturing Agent** запускает холодную последовательность (4 письма: день 1/7/21/45)
4. **Bitrix24 sync:** лид автоматически создаётся как сделка в Bitrix24 (двусторонняя синхронизация)

### Преимущества metadata для CHAMP

| Поле metadata | Влияет на CHAMP-dimension | Как |
|---------------|--------------------------|-----|
| `has_security: true` | Challenges (+) | Есть бюджет на безопасность, знают проблему |
| `has_skud: false` | Technical Readiness (+) | Нет конкурирующей системы |
| `object_size: 120` | Money (+) | Большой объект = достаточный бюджет |
| `management_type: tsn` | Authority (+) | Один ЛПР, быстрое решение |
| `contact_role` | Authority (+) | Подтверждение полномочий |

## Этап 8. Обратная связь и итерация

**Цель:** Улучшать качество парсинга на основе результатов продаж.

### Метрики для отслеживания

| Метрика | Источник | Цель |
|---------|----------|------|
| Конверсия parsed → qualified (CHAMP ≥40) | AI Sales | >60% |
| Конверсия parsed → hot (CHAMP ≥70) | AI Sales | >20% |
| Конверсия parsed → won | Bitrix24 | >5% |
| Среднее quality_score по батчу | Парсер | >0.6 |
| % дубликатов с существующей базой | Bitrix24 | <10% |

### Цикл улучшения

1. Экспорт батча → AI Sales / Bitrix24
2. Через 2-4 недели: выгрузка результатов (won/lost/no response)
3. Анализ: какие источники дают лучшую конверсию?
4. Корректировка: фильтры, приоритеты источников, пороги quality_score
5. Повторный парсинг с улучшенными параметрами

## Рекомендуемая последовательность запуска

### Фаза 1: Валидация (1-2 недели)
1. Парсинг КП Московской области из 2GIS

3. Экспорт в CSV → ручной импорт в Bitrix24

5. **Критерий перехода:** >70% контактов актуальны (телефон/email работают)

1. Добавить Яндекс Карты + парсинг сайтов
2. Подключить webhook в AI Sales Factory
3. Параллельный экспорт: CSV в Bitrix24 + webhook в AI Sales
4. Сравнить конверсию ручной обработки vs AI-обработки
5. **Критерий перехода:** AI-обработка даёт конверсию ≥ ручной
